# Step 1 - Research And Initial Architectural Reasoning

My initial thought was to use C#/TypeScript with SQLite and a full text search extension (FTS5), it would be portable/simple and give decent relational search with fuzzy matching and rankings.

I can also easily convert the provided SQL file data dump to SQLite, if time permits I would prefer to analyse the MIME files against the schema used in the dump and ensure all metadata is contained, otherwise building my own database from the MIME files, but a quick analysis and it look good.

The downsides to an SQL+extensions only option is the fuzzy matching is limited to wildcards with no mis-spelling or similarity detection on words and related email rankings have no semantic relevance, only word approximiation algorithms via FTS5.

Some of this could be overcome by building a tokenized trigram index table and/or query transformations but the trigram index provides decent but non-semantic results, and the latter is not a precomputed index and adds word algorithm computations and requires complex SQL at query time.

I was familiar with using elasticsearch and realised it provided most of these features and more, researching what it used under the hood led me to Lucene which has wrappers for many complex semantic search algorithms built in - I just need to build an index file that points to the SQLite rows and then I have a full text search with powerful semantic + relational queries.

I also considered a PostgreSQL + extensions option which could do everything that lucene+sqlite could, however would be a lot more complicated to configure and would require setting up a dedicated independant service.

As for the language I did consider C# due to familiarity but decided to use Kotlin, although it will take a bit more time to set up a dev environment and get familiar, it's a good time as any to learn, also the C# lucene library is out-dated and lucene is JVM native anyway.

So after analyzing a few different approaches Kotlin+SQLite+Lucene seemed to provide the best user search experience fit for the use case, also being portable and providing a good development opportunity.

This aligns well with stakeholder interest and technical/other resource constraints, for a legal use case the relational + semantic search provides the best case analysis context, especially when presented along with a configurable UI with a data view that can easily be cross referenced, cited and exported.

## Reasoning Re-evaluations

On further analysis of the mysql email data set it is missing some metadata which would be useful in a legal analysis context:

- Attachment metadata in `Content-Disposition` MIME headers
- mysql `Referenceinfo` unable to fully reconstruct email threads compared to MIME threading headers
- `folder` column doesn't contain original header sources
- `date` column in SQL doesnt contain timezone detail when MIME dates generally do
- encodings from `Content-Type` headers are not preserved, mysql dump is old and may have incorrectly processed non-ASCII characters

Building a pre-processing data set parser using Python and its built in email module will probably save any troubles with the mysql source:
- it should match the MIME files as much as possible but still preserve good relational data
- hybrid approach incorporating the mysql dump for the `Employeelist` table as this would be rather complex to reconstruct

Further consideration has also been made on the UI/UX:
- a HTTP API container with a React/TypeScript frontend with autogenerated DTOs / client using OpenAPI specs
- the base UI components have been described below and I have it visualised in my head, time permitted I may do wireframe prototypes

This is what I know well but I may instead explore UI development in Kotlin if time permits.

## Components

### Email Dataset parser

- Parsing class - initializes database with schema, parses the emails and inserts them into the database
- Email and SQL table models (Python dataclasses)

### SQLite Database (datastore)

- Database class - base SQLite data wrapper for database configuration, lifecycle management and core usage
- Table interfaces - interfaces/models for representing the database tables


### Lucene Index (datastore)

- Lucene class - index build function + base search functionality

### Search

- Configuration class - application configuration
- Search class / search core - SQLite + lucene wrapper for core application search function with application aware params, handles core search, email detail retrieval, thread reconstruction + related search (with pagination),
- Search class / input parser - handles input and query parsing for boolean operations and fuzzy matching
- Search class / exporter - handles result export to CSV or a PDF/DOCX with a table
- Search models/interfaces - search result models (list summaries and full detail models)

### CLI

- CLI class - entry point for admin commands: Lucene index rebuild, data validation, status checks

### API

- API class - endpoint handlers for search, email detail, thread view, related, export, and health
- DTO Interfaces - list vs detail DTOs, used by API class and synced via frontend (via OpenAPI)

### UI

- network wrapper class - uses generated DTO via generated OpenAPI+tRPC client, translating between the indexed datastore and requests/responses from the user interface

- UI components
    - Application
    - Search input container
        - Search box (with built in interactive legend/advanced)
        - Search config (toggles/other based on backend Search class params)
    - Search results container
        - Data view - filterable/sortable list/table
        - Data view config (toggles/other for data display, could be backend Search class params or client processing)
        - Email detail view - full email view with all metadata, recipients, attachments
        - Thread view - conversation thread display showing all emails in a thread chain
        - Related view - card for related items
